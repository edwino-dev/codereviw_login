📋 Resumen Ejecutivo

El proyecto es un sistema de autenticación en PHP 8.3 + MySQL con flujo de registro, login, dashboard y logout. El código demuestra un nivel intermedio-avanzado: usa PDO con prepared statements, CSRF tokens, bcrypt con cost 12, y separación básica de responsabilidades en carpetas. Sin embargo, presenta vulnerabilidades de seguridad críticas que deben corregirse antes de cualquier despliegue.

El refactoring debe priorizar: (1) corrección de la sesión post-login, (2) eliminación del anti-patrón de sanitización en entrada, (3) separación MVC, (4) manejo de race conditions.

🔴 Hallazgos de Seguridad
Crítico
Session Fixation — falta session_regenerate_id()
login.php · registro.php
▾

Tras un login exitoso, el ID de sesión no se regenera. Un atacante que obtenga el ID de sesión antes del login (p.ej. via XSS, red compartida o cookie injection) puede secuestrar la sesión autenticada del usuario.

❌ login.php actual — línea ~30
// Después del login exitoso:
$_SESSION['usuario_id']     = $user['id'];
$_SESSION['usuario_nombre'] = $user['nombre'];
// ⚠️  El session ID no cambia — misma sesión pre y post-login
header("Location: dashboard.php");
exit;
✅ Corrección
session_regenerate_id(true); // true = borrar sesión anterior
$_SESSION['usuario_id']     = $user['id'];
$_SESSION['usuario_nombre'] = $user['nombre'];
header("Location: dashboard.php");
exit;
✅ Fix: 1 línea en login.php y registro.php
Crítico
Sanitización de datos en entrada (anti-patrón)
includes/funciones.php · obtenerPost()
▾

obtenerPost() aplica htmlspecialchars() y strip_tags() a los valores POST antes de almacenarlos. Esto corrompe datos legítimos (p.ej. nombre O'Brien → O'Brien en DB) y genera doble-escape al mostrarlos. La sanitización debe ocurrir en la salida, no en la entrada.

❌ Problema
function obtenerPost(string $campo): string
{
    // htmlspecialchars en entrada → datos corruptos en BD
    return isset($_POST[$campo])
        ? htmlspecialchars(trim(strip_tags($_POST[$campo])), ENT_QUOTES, 'UTF-8')
        : '';
}
✅ Corrección — limpia sin escapar para HTML
function obtenerPost(string $campo): string
{
    // Solo trim + strip_tags; htmlspecialchars va al RENDERIZAR, no aquí
    return isset($_POST[$campo])
        ? trim(strip_tags($_POST[$campo]))
        : '';
}
✅ Fix: modificar obtenerPost() — 2 líneas
Alto
Credenciales por defecto en código (root / sin contraseña)
includes/bd.php
▾

Si el archivo .env no existe, el sistema usa root sin contraseña. Cualquier exposición accidental del fuente (GitHub, logs) revela credenciales de acceso a la DB. Además, no se valida que las constantes definidas tengan valores no vacíos.

❌ Fallbacks peligrosos
define('DB_USER', defined('DB_USER') ? DB_USER : 'root');
define('DB_PASS', defined('DB_PASS') ? DB_PASS : '');
✅ Forzar .env en producción
if (!file_exists(__DIR__ . '/../.env')) {
    // En producción: fallar rápido si no hay .env
    error_log('Archivo .env no encontrado');
    http_response_code(503);
    die('Error de configuración del servidor.');
}
✅ Fix: añadir .env.example + validación de entorno
Alto
Logout incompleto — cookie de sesión persiste
public/logout.php
▾

session_destroy() elimina los datos en servidor pero el cliente conserva la cookie con el ID. Si hay un breve window de re-uso del ID (p.ej. antes de que expire el garbage collector), la sesión puede reutilizarse. También falta session_unset().

❌ logout.php actual
session_start();
session_destroy();  // Cookie persiste en cliente
header("Location: index.php");
exit;
✅ Logout robusto
session_start();
session_unset();
session_destroy();
// Eliminar cookie del navegador
setcookie(session_name(), '', time() - 3600, '/', '', true, true);
header("Location: index.php");
exit;
✅ Fix: 2 líneas adicionales en logout.php
Medio
Sin protección contra fuerza bruta en login
public/login.php
▾

No existe rate limiting, captcha, ni bloqueo por intentos fallidos. Un atacante puede realizar ataques de diccionario de forma ilimitada contra cualquier email conocido.

Recomendación mínima para refactoring: registrar intentos fallidos en DB (tabla login_attempts) y bloquear IP/email por N minutos tras X fallos. Alternativa rápida: integrar reCAPTCHA v3 en el formulario.

🐛 Bugs Funcionales
Alto
session_start() llamado DESPUÉS de require_once
public/dashboard.php — línea 2 vs 3
▾

funciones.php se carga antes de session_start(). Si en el futuro funciones.php accede a $_SESSION, el orden incorrecto generará errores silenciosos o warnings. El estándar es iniciar la sesión lo primero.

❌ Orden actual en dashboard.php
require_once __DIR__ . '/../includes/funciones.php';
session_start();  // ← debería ir primero
✅ Orden correcto
session_start();  // ← siempre lo primero
require_once __DIR__ . '/../includes/funciones.php';
Alto
Flash de bienvenida nunca se muestra en Dashboard
public/login.php → public/dashboard.php
▾

Tras login exitoso, login.php llama a flash('success', "¡Bienvenido...") y redirige a dashboard.php. Pero dashboard.php nunca llama a mostrarFlash() — el mensaje se guarda en sesión y nunca se renderiza.

✅ Añadir en dashboard.php
require_once __DIR__ . '/../includes/flash.php';
// ...en el HTML:
<?php mostrarFlash(); ?>
Medio
Race condition en verificación de email único
public/registro.php
▾

El flujo actual hace un SELECT para verificar duplicado y luego un INSERT. Dos requests concurrentes con el mismo email pueden pasar ambos el SELECT y causar un duplicado (violando la constraint UNIQUE de la DB, lo que lanza una excepción no manejada específicamente).

✅ Confiar en la constraint UNIQUE y atrapar la excepción
try {
    $hash = password_hash($pass, PASSWORD_DEFAULT, ['cost' => 12]);
    $stmt = $pdo->prepare("INSERT INTO usuarios (nombre, email, contrasena_hash) VALUES (?, ?, ?)");
    $stmt->execute([$datos['nombre'], $datos['email'], $hash]);
    // Éxito — sin SELECT previo
} catch (PDOException $e) {
    if ($e->getCode() === '23000') { // Duplicate entry
        $errores[] = 'Ese correo ya está registrado.';
    } else {
        error_log("Error en registro: " . $e->getMessage());
        $errores[] = 'Error al crear la cuenta. Intenta más tarde.';
    }
}

▾

💬 Feedback General
👍

Lo que está bien — no romper en el refactoring

PDO con prepared statements y ATTR_EMULATE_PREPARES = false protegen correctamente contra SQL injection. bcrypt con cost 12 es una buena elección. CSRF tokens con random_bytes(32) es sólido. htmlspecialchars() en output protege contra XSS. La estructura básica de carpetas es coherente.

⚠️

Prioridad inmediata (antes de cualquier deploy)

1) Agregar session_regenerate_id(true) post-login. 2) Corregir logout.php para eliminar la cookie. 3) Corregir el orden session_start() en dashboard.php. 4) Eliminar htmlspecialchars() de obtenerPost(). 5) Reemplazar credenciales hardcodeadas con validación estricta de .env.

🗺️

Hoja de ruta del refactoring (orden sugerido)

Sprint 1: fixes de seguridad críticos (listados arriba) + typo + flash en dashboard.
Sprint 2: separar funciones.php en 3 archivos + extraer partials HTML + agregar .htaccess.
Sprint 3: implementar MVC básico + Composer + primeros tests PHPUnit.
Sprint 4: rate limiting + reset de contraseña + verificación de email.

🎯

Nivel general del código

El código es claramente mejor que un tutorial básico — hay intención de aplicar buenas prácticas. Los problemas son de madurez y de detalles de implementación, no de base. Con el refactoring propuesto, el proyecto puede alcanzar un nivel sólido de producción.

Documentan las decisiones arquitectónicas actuales y propuestas para el refactoring.

ADR-001 Uso de PDO con prepared statements para acceso a DB Aceptado
FechaMar 2026
ContextoAcceso a MySQL desde PHP
ImpactoAlto
RevisiónNo necesaria

Contexto: El sistema necesita consultar y escribir en MySQL. PHP ofrece mysqli nativo, PDO, y ORMs.

Decisión: Se usa PDO con prepared statements reales (ATTR_EMULATE_PREPARES = false) y FETCH_ASSOC como modo por defecto.

Consecuencias positivas:

  • Prevención completa de SQL injection sin esfuerzo adicional
  • API portátil (cambia el driver, no el código)
  • Manejo de errores vía excepciones (ERRMODE_EXCEPTION)

Consecuencias negativas: Sin capa Model, las queries están dispersas en los controladores.

ADR-002 bcrypt (PASSWORD_DEFAULT) con cost 12 para contraseñas Aceptado
FechaMar 2026
ContextoAlmacenamiento de contraseñas
ImpactoCrítico
RevisiónAnual (ajustar cost)

Decisión: password_hash($pass, PASSWORD_DEFAULT, ['cost' => 12]) con límite de 72 chars (límite de bcrypt).

Alternativas consideradas: Argon2id (más moderno) — descartado para mantener compatibilidad con PHP 7.x. En PHP 8.3 podría migrarse a Argon2id en el futuro.

  • Cost 12 = ~250ms por hash en hardware moderno — aceptable para UX y resistente a GPU cracking
  • PASSWORD_DEFAULT garantiza upgrade automático si PHP cambia el algoritmo
ADR-003 Adoptar arquitectura MVC mínima en el refactoring Propuesto
FechaMar 2026
ContextoRefactoring del proyecto
ImpactoAlto
DecisorEquipo de desarrollo

Contexto: Actualmente la lógica de negocio, el acceso a DB y el HTML están en los mismos archivos, dificultando el testing y la evolución del código.

Decisión propuesta: Separar en Controllers/ (lógica), Models/ (DB) y Views/ (HTML). Sin framework externo — MVC "casero" con Composer + PSR-4.

Opciones evaluadas:

  • Micro-framework (Slim, Laravel): mayor curva de aprendizaje, overhead para proyecto pequeño
  • MVC manual con Composer: ✅ elegido — control total, sin dependencias pesadas
  • Mantener estructura actual: deuda técnica creciente, dificulta testing

Consecuencias: Refactoring de 2-3 sprints. Permite agregar tests unitarios desde el Sprint 3.

ADR-004 Sanitización en salida, no en entrada (output encoding) Propuesto
FechaMar 2026
ContextoCorrección de anti-patrón en obtenerPost()
ImpactoMedio-Alto
UrgenciaSprint 1

Contexto: obtenerPost() actualmente aplica htmlspecialchars() a los datos antes de guardarlos en DB — corrompe datos con caracteres especiales y crea doble-escape en el output.

Decisión: Adoptar el principio OWASP de "sanitize on output": los datos se guardan en DB tal como vienen (tras trim y strip_tags); htmlspecialchars() se aplica solo al momento de renderizar en HTML.

  • Los datos en DB son fieles a lo que el usuario ingresó
  • El encoding correcto depende del contexto de salida (HTML, JSON, CSV — cada uno tiene su escape)
  • Elimina bugs de doble-escape silenciosos
ADR-005 Gestión de configuración exclusivamente via .env Propuesto
FechaMar 2026
ContextoCredenciales y configuración de entornos
ImpactoCrítico
UrgenciaSprint 1

Decisión: Eliminar todos los valores por defecto de credenciales en bd.php. El sistema debe fallar explícitamente si .env no existe en producción. Agregar .env.example al repositorio y .env al .gitignore.

  • Seguir el principio 12-Factor App (config en entorno, no en código)
  • Previene exposición accidental de credenciales en VCS
  • Facilita el manejo de múltiples entornos (dev/staging/prod)
